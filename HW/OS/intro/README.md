**Эти требования обязательны для всех домашних заданий в течение семестра.**

Используйте [Google Style Guide](https://google.github.io/styleguide/cppguide.html) — в той части, в которой он актуален для C. Вы можете запустить форматирование и проверки командой `clang-format -Werror */*.c */*.h`.

## Часть 3. Pingpong

Прежде чем перейти к основной части курса, познакомимся подробнее с xv6 и её системными вызовами. Мы попробуем написать немного user-space кода.

Научитесь обмениваться данными между процессами с помощью специальных FIFO-каналов — [Unix pipes](https://en.wikipedia.org/wiki/Pipeline_(Unix)). Реализуйте программу `user/pingpong.c`. Она должна:

* Создать пайпы для связи двух процессов.
* Форкнуться — создать дочерний процесс.
* Родительский процесс должен отправить сообщение `ping` в дочерний с помощью пайпы.
* Дочерний процесс должен прочитать это сообщение и вывести `<child pid>: got <message>`. После окончания сообщения он должен отправить `pong` родителю.
* Родительский процесс должен прочитать ответ и вывести `<parent pid>: got <message>`.
* Оба процесса должны успешно завершиться.

Советы для выполнения задания:

* Вам понадобится несколько системных вызовов — `pipe`, `fork`, `read`, `write`, `getpid`. Воспользуйтесь утилитой `man`, чтобы узнать, что делают эти вызовы и как ими пользоваться — поведение в xv6 не особо отличается от других Unix-подобных операционных систем.
* Сигнатуры всех доступных системных вызовов приведены в [`user/user.h`](user/user.h).
* Вместо привычных вам `stdlib.h` и `stdio.h` доступна местная стандартная библиотека — [`user/ulib.c`](user/ulib.c), [`printf.c`](user/printf.c) и [`umalloc.c`](user/umalloc.c). Посмотрите на другие программы в директории [`user/`](user/), чтобы понять, как ей пользоваться.
* Добавьте программу в `UPROGS` в `Makefile`, чтобы она скомпилировалась.
* Ваша программа должна быть универсальной: «читающий» процесс не должен знать ничего о длине передаваемой ему строки.
* В программах для xv6 обязательно нужно вызывать `exit(0)` для выхода.

## Часть 4. Reg

В прошлом подзадании мы использовали системные вызовы, например, `pipe` и `fork`. Задача системных вызовов — дать программам из user-space возможность выполнять «привилегированные» команды.

Ваша задача — реализовать новый системный вызов `reg`. Этот системный вызов должен уметь получать значения регистров `s2`—`s11` указанного процесса.

Он должен принимать три аргумента:
* `int pid` — номер процесса, для которого запрашивается значение регистра
* `int register_num` — номер регистра, число от 2 до 11
* `uint64 *return_value` — адрес, по которому необходимо записать значение

В целях безопасности процесс может запрашивать только свои регистры или регистры дочерних процессов. При этом доступны лишь непосредственные дети этого процесса, а не всё дерево процессов.

Начните с добавления функции `reg` в файл [`kernel/proc.c`](kernel/proc.c). Текущий процесс можно получить с помощью функции `myproc`, а все процессы находятся в массиве `proc`. Структура `proc` содержит поле `trapframe`, в котором находятся значения всех регистров. Избегайте лишней копипасты при получении регистра.

Аргументы в системные вызовы приходят особым образом. Изучите файл [`kernel/sysproc.c`](kernel/sysproc.c), чтобы посмотреть, как обрабатываются аргументы в других системных вызовах. Аналогичным образом отредактируйте файлы [`kernel/syscall.h`](kernel/syscall.h), [`kernel/sysproc.c`](kernel/sysproc.c) и [`kernel/syscall.c`](kernel/syscall.c) так, чтобы появилась возможность вызвать `dump` из user-space.

Чтобы системный вызов был доступен из user-space, добавим в файл [`user/user.h`](user/user.h) объявление функции `reg`, как это сделано для других системных вызовов. В файл [`user/usys.pl`](user/usys.pl) добавьте строку `entry("reg")` — он отвечает за генерацию ассемблерных инструкций для совершения системного вызова.

Осталось собрать xv6. Запустите утилиту [`regtests`](user/regtests.c), чтобы проверить работу вашего системного вызова.

* Функцию `reg` нужно также определить заголовочном файле в [`kernel/defs.h`](kernel/defs.h).
* Системный вызов должен возвращать 0 при успешном завершении, и код ошибки в остальных случаях. Используйте следующие коды ошибок:
   * `-1` — вызвавший процесс не имеет прав на получение запрошенного регистра
   * `-2` — процесса с таким идентификатором не существует
   * `-3` — некорректный номер регистра
   * `-4` — не удалось записать данные по указанному адресу
* Вы не можете записать данные напрямую по адресу `*return_value` — это виртуальный адрес в user-space, и использовать его в kernel-space невозможно. Вам поможет функция `copyout`.

## Часть 5*. Xargs (+3 балла)

> Это — необязательная часть. Если вы выполните её, то получите бонусные баллы. Помните, что необязательная часть сдаётся только вместе со всем заданием — досдать её отдельно не получится.

Реализуйте ещё одну утилиту в user-space — `xargs`. 

UNIX-утилита `xargs` позволяет запускать какую-то команду несколько раз по шаблону, подставляя аргументы из входного файла. Например, попробуйте в вашем терминале выполнить:

```bash
$ echo /tmp | xargs ls -la
$ # Запустилось `ls -la /tmp`
drwxrwxrwt  2 root root   4096 Sep 01 00:00  .X11-unix
drwxrwxrwt  2 root root   4096 Sep 01 00:00  .XIM-unix

$ (echo a; echo b) | xargs -n 1 mkdir
$ # Запустилось `mkdir a` и `mkdir b`
$ ls -la
drwxrwxr-x  2 user user 4096 Sep 01 00:00 a
drwxrwxr-x  2 user user 4096 Sep 01 00:00 b
```

> Настоящий `xargs` умеет оптимизировать вызовы, подставляя сразу несколько аргументов в вызов, например, во втором примере
> `(echo a; echo b) | xargs mkdir` запустил бы `mkdir a b`. Флаг `-n 1` отключает такое поведение, заставляя передавать
> по одному параметру в каждый вызов. Вам нужно реализовать именно такой вариант поведения.

Для каждой строки входного файла выполните команду, переданную аргументами в xargs, подставив содержимое строки в конец.

Вам понадобятся системные вызовы `fork`, `exec` и `wait`. Последний нужен, чтобы дождаться выполнения дочернего процесса.

Полезные советы:
* В `kernel/param.h` есть переменная `MAXARG`. Считайте, что больше аргументов вам передавать не придётся.
* Файловая система не пересоздаётся при каждом запуске QEMU. `make clean` удалит ФС и последующий вызов `make qemu` создаст чистую ФС.
